# 1. Solid design phase

Do not start coding the sidebar overhaul directly. First lock these design decisions.

## A. Product goals

The new sidebar/search UX must achieve four things:

1. make search feel like navigation, not a tool
2. make intent visible, not hidden in logs
3. make refinement easy after results appear
4. keep background AI work visible and non-blocking

If a change does not improve one of those four, it is probably decoration.

## B. UX principles

Use these as hard rules:

### Rule 1, one control area, one result area

Left side controls intent and refinement.
Center shows results.
Top shows current query state.

### Rule 2, search is stateful

A search is not just text. It is:

* query text
* active preset
* active family
* active filters
* active sort
* result count

This state must be visible in UI.

### Rule 3, refine after search

Do not show every filter all the time.
Show contextual refinements only after:

* a preset was selected,
* a query was typed,
* or a result family is active.

### Rule 4, discoverability beats raw power

Google Photos wins because users can discover features without understanding the system.
Lightroom wins because experts can still control it.
You need both.

### Rule 5, one canonical search state store

Do not let:

* sidebar,
* top bar,
* results header,
* smart find cards,
* people section

each maintain their own local search truth.
There must be one shared state object.

## C. UX information architecture

The left sidebar should have exactly these sections:

### Search

* search input
* recent searches
* suggestions
* active query chips

### Discover

* smart presets as visual cards
* Documents
* Screenshots
* People
* Places
* Scenic categories

### People

* top person clusters
* merge-review candidates
* unnamed clusters

### Browse

* All Photos
* Albums
* Folders
* Dates
* Favorites
* Videos
* With Location

### Filters

Hidden until a search/preset is active. Then show:

* People
* Location
* Time
* Type
* Rating / Favorite
* Camera / Lens, later

### Activity

Compact background-task strip:

* scanning
* embeddings
* face detection
* clustering
* OCR later

## D. User flows to design before coding

You need mock flows for these seven actions:

1. user opens app with no project
2. user opens app with active project
3. user clicks Beach preset
4. user searches “kids beach 2024”
5. user clicks Face_007 in People
6. user applies filters after results
7. user sees background indexing / face clustering progress

If any of these flows is awkward on paper, code will not fix it.

# 3. State architecture, exact recommendation

Use one central store.

## `SearchStateStore`

Suggested dataclass:

```python
from dataclasses import dataclass, field
from typing import Optional, List, Dict, Any

@dataclass
class SearchState:
    query_text: str = ""
    preset_id: Optional[str] = None
    family: Optional[str] = None
    active_filters: Dict[str, Any] = field(default_factory=dict)
    active_people: List[str] = field(default_factory=list)
    active_chips: List[Dict[str, Any]] = field(default_factory=list)
    sort_mode: str = "relevance"
    result_count: int = 0
    search_in_progress: bool = False
    has_active_project: bool = False
    active_project_id: Optional[int] = None
    warnings: List[str] = field(default_factory=list)
```

Use Qt signals in a QObject wrapper:

```python
class SearchStateStore(QObject):
    stateChanged = Signal(object)

    def __init__(self):
        super().__init__()
        self._state = SearchState()

    def get_state(self) -> SearchState:
        return self._state

    def update(self, **kwargs):
        for k, v in kwargs.items():
            setattr(self._state, k, v)
        self.stateChanged.emit(self._state)
```

## `SearchController`

Responsibilities:

* receives UI events
* translates them into orchestrator requests
* updates store
* coordinates search execution
* handles stale-result cancellation

This is where all search actions should go.

# 7. Critical design warnings

## Warning 1

Do not let each section query DB independently whenever possible.
Use controller/state-driven refresh, or you will recreate the same inconsistency problems you just spent days fixing.

## Warning 2

Do not embed business logic in widgets.
Widgets render and emit signals.
Controllers decide.

## Warning 3

Do not expose all filters all the time.
That becomes “enterprise software ugly” fast.

## Warning 4

Do not make the left sidebar purely static.
It must reflect current search intent.

# 8. What good looks like

After this redesign, the experience should feel like:

* type search in one place
* see why results are shown
* refine with chips and contextual facets
* jump between People / Discover / Browse without losing search state
* always know what the app is doing in background

That is much closer to Google Photos / Apple Photos / Lightroom / Excire-grade UX than your current likely mixed sidebar.
